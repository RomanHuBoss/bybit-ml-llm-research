# UI audit: intraday operator cockpit

## Проверенные проблемы предыдущей компоновки

- Панели были визуально равноправными, хотя по смыслу часть из них является первичным решением, часть — доказательствами, часть — журналом.
- Правая колонка создавала «кашу» из risk gate, evidence, LLM, news и protocol: все блоки одновременно конкурировали за внимание оператора.
- Технические данные занимали слишком заметное место относительно главного вопроса: можно ли вручную рассматривать вход или вход запрещен.
- MTF, trade ticket и risk gate были связаны слабо: оператору приходилось глазами собирать решение из нескольких независимых областей.

## Принятое UX-решение

Фронтенд теперь построен как двухзонный cockpit:

1. Левая зона — очередь кандидатов и операции с данными.
2. Основная зона — первичное решение, trade sheet, MTF-consensus и вкладки контекста.

Постоянно видимыми оставлены только сущности, которые нужны для первичного решения:

- статус решения;
- score;
- entry / SL / TP / R/R;
- MTF 15m/60m/240m;
- выбранная вкладка проверки.

Вторичные сущности вынесены во вкладки:

- Risk;
- Evidence;
- LLM;
- News;
- Protocol.

Технические данные, equity chart, журнал и raw table остаются доступными, но скрыты в collapsible-блоке.

## Safety-инварианты UI

- `LIVE OFF` всегда видим в верхней панели.
- Красный пункт risk gate трактуется как запрет входа.
- 60m/240m остаются контекстом и veto-фильтрами, а не самостоятельными entry-trigger.
- LLM явно маркируется как риск-разбор, а не торговый приказ.
- Операции загрузки/пересчета данных вынесены в drawer, чтобы не конкурировать с решением.

## Исправление компоновки блока операций с данными

Блок `Операции с данными` закреплен как нижняя зона левого rail и больше не зависит от количества карточек в очереди кандидатов. Левая колонка теперь имеет три независимые зоны:

1. KPI-сводка — фиксированная верхняя зона.
2. Очередь кандидатов — единственная область с внутренней прокруткой по списку рекомендаций.
3. Операции с данными — всегда доступный нижний drawer.

При раскрытии drawer его содержимое получает собственную прокрутку. Это предотвращает ситуацию, когда операции загрузки, пересчета и ML/LLM-кнопки уходят ниже viewport при большом количестве рекомендаций или при переключении фильтров очереди.

## Исправление scroll-контейнера блока «Операции с данными»

Блок операций не должен зависеть от длины очереди рекомендаций и выбранной вкладки проверки. Левая колонка построена как flex-контейнер: KPI имеет естественную высоту, очередь кандидатов занимает оставшееся место и прокручивается внутри себя, а drawer операций закреплен как нижняя рабочая зона.

Критичный инвариант: внешний `left-rail` может ограничивать общий viewport, но не должен использоваться как рабочая область прокрутки. Прокрутка локализована в двух местах: `candidateQueue` для списка рекомендаций и `ops-body` для раскрытых операций с данными.

## Исправление раскрытия панели «Операции с данными»

Нативный `details/summary` был заменен на явный контролируемый drawer `opsPanel` с кнопкой `opsToggleBtn`. Причина: в sticky/flex layout нативный `details` оказался хрупким — при комбинации `overflow`, `max-height`, `flex-basis` и малой высоты viewport панель могла визуально не раскрываться или раскрывалась без доступной области прокрутки.

Новый инвариант:

- свернутая панель всегда видна как нижняя кнопка левого rail;
- раскрытая панель получает класс `open`, `aria-expanded=true`, а тело `opsBody` снимается с `hidden`;
- на desktop раскрытая панель остается в нормальном потоке левого rail и не использует `position: fixed`, чтобы не наезжать на очередь рекомендаций;
- содержимое операций прокручивается внутри `opsBody`;
- клавиша `Escape` закрывает панель;
- на узких экранах панель возвращается в обычный поток документа без вложенного fixed-дисплея.


## Исправление наезда панели операций на очередь кандидатов

Предыдущая версия фиксировала раскрытый drawer через `position: fixed`. Это устраняло clipping, но создавало новый дефект: панель операций могла перекрывать `Candidate queue`, особенно при низкой высоте окна.

Новый инвариант:

- `opsPanel.open` остается в нормальном flex-потоке левой колонки (`position: relative`);
- `queue-panel` и `ops-panel` делят высоту `left-rail`, а не накладываются друг на друга;
- очередь кандидатов получает собственную прокрутку и сжимается при открытии операций;
- тело операций `opsBody` получает собственную прокрутку и ограниченную высоту;
- на desktop запрещены `position: fixed`, `left`, `bottom` для раскрытого блока операций.

## Инвариант intraday-очереди

Очередь решений в operator cockpit показывает только реальные entry-рекомендации по `MTF_ENTRY_INTERVAL` (по умолчанию 15m). Сигналы 60m и 240m не должны отображаться как отдельные карточки кандидатов: они являются bias/regime-контекстом внутри MTF matrix и могут подтвердить, ослабить или запретить 15m-сетап.

Проверяемые правила:

- `candidateQueue` не должен получать `CONTEXT_ONLY` как самостоятельную рекомендацию;
- `/api/research/rank` при включенном MTF возвращает только entry-кандидаты;
- `/api/signals/latest` по умолчанию отдает только entry-сигналы для UI;
- LLM/background backtest не должны тратить очередь на 60m/240m context-only сигналы.

## Дополнение UX/верстки 2026-04-26: redesign pass v2

По дополнительной проверке интерфейс был оценен как недостаточно удобный для реальной работы оператора: главный вердикт конкурировал с техническими блоками, карточки очереди были мелкими, а визуальная иерархия не помогала быстро понять, где запрет входа, где наблюдение, а где сетап можно разбирать вручную.

Исправления:

- переписана визуальная система `frontend/styles.css`: более контрастная dark-тема, крупный `decision-board`, единый card/grid rhythm, более читаемые KPI, chips, badges и trade-ticket;
- очередь кандидатов стала рабочей областью с локальной прокруткой и устойчивым `scrollbar-gutter`, а не бесконечной колонкой, вытесняющей операции;
- drawer операций закреплен в нормальном flex-потоке левого rail и больше не перекрывает очередь;
- блоки `Risk / Evidence / LLM / News / Protocol` визуально разделены как стадии проверки, а не как равнозначные технические панели;
- все кнопки получили `type="button"`, чтобы исключить неожиданный submit при будущей обертке в форму;
- очередь кандидатов получила `aria-live="polite"`, чтобы обновления не были полностью невидимы для assistive-технологий;
- добавлен глобальный busy-state: при длительной операции кнопки блокируются, повторный запуск не стартует параллельную операцию;
- API-запросы фронтенда получили `AbortController` timeout, чтобы UI не зависал бесконечно при подвисшем backend/API;
- внешние URL новостей проходят whitelist `http/https` через `safeExternalUrl`; `javascript:` и другие схемы не попадают в `href`;
- CSS-классы, построенные из backend-данных, нормализуются через `cssToken`, чтобы неожиданные значения не ломали верстку.

Ограничение: это не заменяет полноценный browser-based visual regression suite. Текущая защита — статические regression-тесты структуры/инвариантов UI плюс `node --check` для JS.

## Frontend redesign pass v3: светлый operator cockpit

После визуальной проверки предыдущей dark-версии интерфейс был переработан с нуля как спокойный light fintech cockpit. Цель — убрать «режущую глаза» неоновую техническую стилистику и дать оператору экран, в котором главный вердикт, очередь кандидатов и ручная проверка считываются за несколько секунд.

Что изменено:

- введена новая shell-структура `app-frame`: узкий левый sidebar, sticky topbar, рабочая сетка `left-rail + workspace`;
- темная палитра заменена на светлую enterprise-систему: теплый серо-белый фон, белые карточки, мягкие границы, сдержанные blue/green/red/amber акценты;
- `decision-board` стал главным визуальным объектом экрана: крупный заголовок решения, короткая причина, отдельный badge и score-блок;
- очередь кандидатов стала компактным списком вместо перегруженных карточек: символ, статус, направление, стратегия и score без лишнего шума;
- блок `Операции с данными` по умолчанию раскрыт, но ограничен по высоте и имеет собственную прокрутку; вторичные операции визуально отделены от двух основных кнопок;
- `Trade sheet` теперь выглядит как таблица параметров сделки, а не набор несвязанных плиток;
- `MTF consensus` оформлен как три компактных state-card: entry / bias / regime;
- risk/evidence/LLM/news/protocol сведены в нижний support-блок, чтобы не конкурировать с главным решением;
- технические детали, журнал и сырые данные остались в `<details>` и не мешают операторской работе;
- карточки кандидатов получили keyboard-accessibility: `role="button"`, `tabindex="0"`, выбор по `Enter` и `Space`;
- добавлены regression-проверки светлой дизайн-системы, sidebar-структуры, открытого блока операций и классов решения.

Сохраненные safety-инварианты:

- `LIVE OFF` остается в верхней панели;
- красный пункт risk-gate по-прежнему означает отмену входа;
- система явно сообщает, что не создает бота автоматически;
- 60m/240m остаются MTF-контекстом/veto, а не отдельными entry-рекомендациями;
- timeout API, busy guard, whitelist внешних ссылок и нормализация CSS-токенов сохранены.

## Frontend corrective pass v4: проверка кнопок, панелей и снижение яркости

Повторная ручная проверка интерфейса выявила два класса проблем: часть видимых контролов выглядела интерактивной, но фактически не имела обработчиков, а светлая палитра получилась слишком яркой для длительной операторской работы.

Исправлено:

- sidebar больше не является декоративным: каждая кнопка имеет `data-nav-target` и выполняет действие — переход к рабочей области, открытие графика бэктеста, переход к protocol, раскрытие блока операций, фокус на настройках или открытие help-dialog;
- кнопка burger `navToggleBtn` теперь реально сворачивает/разворачивает sidebar через класс `nav-collapsed`;
- добавлен `helpDialog` с краткой операторской инструкцией и рабочей кнопкой закрытия;
- добавлен видимый статус операций `operationToast` сверху и `operationStatus` внутри блока операций — ошибки больше не прячутся только в техническом журнале;
- `runOperation()` больше не пробрасывает исключения в консоль как unhandled promise rejection: ошибка фиксируется в журнале и явно показывается оператору;
- busy-state больше не блокирует весь экран. Во время долгого API-запроса disabled получают только кнопки с `data-busy-lock="true"`; навигация, вкладки, фильтры, help и сворачивание панелей остаются рабочими;
- автообновление не вмешивается в уже выполняющуюся ручную операцию: periodic refresh пропускает цикл при `is-busy`;
- фильтры очереди и context-tabs синхронизируют `aria-selected`, а не только CSS-класс;
- панели context-tabs получают `hidden`, поэтому неактивные секции не остаются доступными для assistive-технологий;
- добавлена базовая клиентская валидация `validateInputs()` перед API-операциями, чтобы пустые символы/интервалы и невалидные дни истории не уходили на backend;
- палитра заменена с холодного ярко-белого/синего на приглушенную warm-neutral: `#ece8df`, `#fbfaf6`, `#f4f1ea`, с muted blue `#3f5f85` вместо насыщенного bright-blue;
- canvas equity-графика приведен к той же приглушенной палитре и защищен от отсутствующего 2D context.

Проверенные инварианты:

- все видимые кнопки либо имеют явный JS-обработчик, либо являются нативным submit внутри `dialog`, либо работают через общий делегированный обработчик `data-nav-target`/`data-tab`/`data-filter`;
- операции с API остаются single-flight, но не делают интерфейс «мертвым»;
- ошибки операций видны без раскрытия технических деталей;
- технические детали остаются вторичным блоком и раскрываются при переходе к графику.

## UI v5 — пересборка после повторной проверки элементов

В этой версии фронтенд сверстан заново по более строгой сетке:

- `Candidate Queue` больше не находится в схлопнутом/усеченном состоянии: карточка закреплена как полноценная рабочая таблица с заголовками, табами, видимыми строками, score и статусами.
- `Операции с данными` вынесены в отдельную полноширинную accordion-панель под основными торговыми карточками. Заголовок панели является реальной кнопкой, `aria-expanded` синхронизируется с состоянием, тело скрывается через `.ops-panel:not(.open) .ops-body`.
- Главная плашка решения уменьшена: вместо огромного баннера используется компактная карточка `ПЕРВИЧНОЕ РЕШЕНИЕ` с символом, вердиктом, причиной, badge и score.
- Цветовая система заменена на спокойную светлую B2B/fintech-палитру: белые карточки, теплый серый фон, приглушенный синий акцент, мягкий зеленый/янтарный/красный для состояний.
- Все основные видимые элементы получили явные affordance-сигналы: табы, refresh-кнопки, кликабельные строки очереди, accordion-toggle, action-кнопки операций, ссылки нижних карточек.
- Добавлена отдельная кнопка `refreshQueueBtn` для обновления очереди без полной перезагрузки всех виджетов.
- `decisionVerdict` отделен от `decisionTitle`, чтобы символ и решение не слипались в один огромный заголовок.

Проверки, выполненные для UI v5:

```text
node --check frontend/app.js — OK
python -S -m compileall -q app tests run.py install.py sitecustomize.py — OK
static frontend checks — OK: уникальные id, корректные button type, наличие ключевых UI hooks
```

Ограничение окружения: полный `pytest` в контейнере периодически зависает при старте окружения; поэтому для UI v5 дополнительно выполнены независимые статические проверки HTML/CSS/JS и синтаксическая проверка JavaScript.

## UI v6 — выравнивание рабочих панелей и дедупликация очереди

Повторная проверка реального экрана выявила два дефекта, влияющих на удобство оператора:

- основные панели `Candidate Queue`, `Trade Sheet`, `MTF Consensus` имели разную высоту, из-за чего экран визуально распадался и нижние блоки начинались на разных уровнях;
- очередь кандидатов могла показывать много строк одной и той же валютной пары, потому что backend возвращает несколько вариантов сетапа/стратегии для одного рынка и entry-таймфрейма.

Исправления:

- для `Candidate Queue`, `Trade Sheet`, `MTF Consensus` введена единая высота через `height: clamp(430px, 45vh, 560px)`;
- переполнение внутри этих трех панелей теперь уходит во внутренний вертикальный скроллинг: `candidate-queue`, `ticket-body`, `mtf-matrix`;
- строка кандидата переразмечена на отдельные колонки `звезда / символ / статус / score / переход`, поэтому статус больше не выдавливает символ в многоточие;
- во frontend добавлена функция `dedupeCandidatesByMarket()`: в очереди остается лучший кандидат на пару `symbol + interval`, остальные варианты учитываются как скрытые дубли;
- выбор лучшего варианта выполняется детерминированно через `compareCandidates()` по приоритету решения, score, confidence, profit factor, drawdown и win rate;
- footer очереди показывает, сколько уникальных рынков отображено и сколько дублей скрыто.

Ограничение: дедупликация сделана на уровне отображения frontend. Backend по-прежнему может возвращать несколько стратегических вариантов одного рынка; это полезно для аудита, но операторская очередь теперь показывает один лучший вариант на рынок/entry-TF.
