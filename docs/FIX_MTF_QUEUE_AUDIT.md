# Fix: MTF enrichment and operator veto logic

Дата ревизии: 2026-04-28

## Что было найдено

1. Визуальный дефект `MTF: не рассчитан` возникал не потому, что MTF-контур всегда не считал 15m/60m/240m, а из-за смешивания двух источников во фронтенде:
   - `/api/research/rank` возвращал обогащенные строки с MTF, liquidity, backtest, ML и LLM;
   - `/api/signals/latest` возвращал сырые сигналы без MTF-контекста;
   - UI предпочитал сырые сигналы, если они были непустыми, поэтому часть строк теряла MTF-поля.

2. Решение `НЕТ ВХОДА` было чрезмерно жестким: отсутствие ML ROC-AUC или отсутствие свежего backtest для конкретного `symbol+TF+strategy` трактовались как hard veto. В результате даже нормальный 15m-сетап с MTF-подтверждением, приемлемым R/R, ликвидностью и spread мог попадать в `НЕТ ВХОДА` только из-за незаполненной доказательной витрины.

## Что исправлено

1. `/api/signals/latest` теперь при включенном MTF загружает entry/bias/regime интервалы, применяет `apply_mtf_consensus()` и возвращает entry-строки уже с MTF-полями.
2. Фронтенд теперь использует `/api/research/rank` как канонический источник operator queue, а `/api/signals/latest` — только как fallback.
3. Отсутствующие ML/backtest/liquidity/spread поля больше не превращаются автоматически в красный hard stop:
   - отсутствие ML-модели = warning;
   - отсутствие backtest = warning;
   - неизвестный liquidity/spread snapshot = warning;
   - hard stop остается для устаревшего сигнала, неверного направления, MTF-конфликта, явно недопущенной ликвидности, слишком широкого spread, невалидного R/R и слишком низкой confidence.
4. `MTF: не рассчитан` на fallback-строках теперь считается warning, а не скрытым pass.

## Ожидаемое поведение после фикса

- Для 15m-кандидатов MTF должен показываться как `15m+60m+240m согласованы`, `15m подтвержден 60m`, `Только 15m`, `Слабая MTF-связь` или `Конфликт старших TF`.
- `MTF: не рассчитан` допустим только как диагностический fallback, если rank/API недоступны или в базе нет достаточного MTF-контекста.
- Отсутствие ML/backtest больше не должно обнулять очередь входов: сетапы будут переходить минимум в `НАБЛЮДАТЬ`, если нет настоящих hard stop-факторов.
- `К ПРОВЕРКЕ` появляется только при отсутствии hard stop и warning-пунктов и достаточном итоговом score.
