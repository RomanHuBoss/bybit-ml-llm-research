# Operator Queue Stability V14

## Проблема

До исправления UI мог создавать впечатление хаотичных торговых рекомендаций: один и тот же symbol/TF попадал в очередь несколькими строками по разным стратегиям, а при наличии близких LONG/SHORT-сценариев выбранная карточка могла меняться после обновления rank/signals/LLM/backtest evidence.

Для советующей СППР это опасно: оператор должен видеть не поток сырых стратегических строк, а устойчивый, объяснимый вердикт по рынку.

## Исправление

Добавлен модуль `app/operator_queue.py`.

Он выполняет серверную стабилизацию:

1. группирует строки по `category + symbol + interval`;
2. сравнивает только последнюю свежую `bar_time` внутри рынка;
3. рассчитывает силу направления по `operator_score`, `confidence`, `mtf_score`, `risk_reward` и уровню решения;
4. возвращает максимум одну строку на рынок;
5. если LONG и SHORT близки по силе — принудительно выставляет `NO_TRADE` / `КОНФЛИКТ СИГНАЛОВ`;
6. если доминирование направления слабое — понижает `REVIEW_ENTRY` до `WAIT`;
7. добавляет диагностические поля `direction_conflict`, `direction_strength_long`, `direction_strength_short`, `direction_gap`, `operator_variant_count`, `operator_stability_score`.

## Frontend

UI теперь:

- сохраняет выбор пользователя по `symbol|interval`, даже если id строки изменился после пересчета;
- показывает stability/conflict в карточке, checklist и explanation panel;
- закрывает панель операций по Escape, а не оставляет ее открытой;
- не превращает дубли стратегий в отдельные визуальные «рекомендации».

## Безопасное допущение

Если система не может доказать единое направление рынка, вход запрещается. Это консервативное поведение намеренно важнее количества рекомендаций.

## Проверки

Добавлены unit-сценарии в `tests/test_operator_queue_stability.py`:

- дубли стратегий одного направления сворачиваются в один рынок;
- близкий LONG/SHORT-конфликт переводится в `NO_TRADE`;
- слабое доминирование направления не остается автоматическим `REVIEW_ENTRY`.
